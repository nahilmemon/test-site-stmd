Dhakira Montage

Problem

Often, the official story told by UNESCO regarding the heritage of a site is not completely accurate and/or is missing information. The Dhakira Heritage Studies Center is trying to create a tool to enable natives of a heritage site to be able to record their own stories regarding their heritage in the form of text, audio, photos, and/or videos. To do so, this center wanted a website and mobile app ecosystem to provide the tools to record and upload their recordings while also providing a platform for natives to create groups with each other and publish content together as a group regarding particular themes. This web/mobile app would allow members to publish their stories privately, within a group setting, or publicly. This app would also double as a research tool for professors and researchers studying about the heritage of a particular site or region.

Solution

Working in a small group of three, we decided to create this app using the meteor framework so that it would be easy to create the mobile apps quickly once the website’s development is complete. I was in charge of most of the front-end web development side as well as the branding identity for the Dhakira Heritage Studies Center. The demo and source code links below only show the front-end side of the project. Work is currently underway to connect the front-end and the backend together and the source code can be found here.

Responsive Images Blog

Problem

In this project, I was given the code for a photo journal style blog which only displayed well on a large screen and had super high resolution images. This made the website

  • look terrible on smaller devices
  • unnecessarily take very long to load (it would take minutes to load on a slow internet connection)

Solution

I optimized the website so that it would load much faster by:

  • Using Grunt and the grunt-responsive-images plugin to create different sizes of each image and at a lower quality
  • Using the picture element with multiple sources so that the browser can decide at runtime which image size to use depending the size of the browser, the size that the image needed to be displayed at, and the pixel density ratio of the user’s devices
  • Using the Zocial font icon to display social media links
  • Using UTF characters instead of PNG files to display emoticons
  • Using an inline SVG element for the logo
  • Using a border instead of the flourish image file

Other ways I tried to improve the website included:

  • Using HTML semantic elements
  • Placing each image in a figure element and giving each image a figcaption and an alt tag so that those using a screen reader or a text based browser could still understand what was in the images
  • Using art direction and tighter crops of some of the images so that they could display better on smaller devices
  • Making the images resize automatically based on the browser width
  • Increase the padding on the links for smaller devices so that they are easier to touch accurately with fingers
  • Add some Google Fonts and a color palette

Responsive Resume Site

Problem

For this project, I was given the code for a resume site for a hypothetical client. This site was designed to only display nicely on a 1926px wide browser. On smaller devices, most of the content would get covered up or go off screen and the text was also very small.

Solution

I made the site more responsive by:

  • Adding the viewport meta tag so that the viewport’s device-width is set properly
  • Changing the layout of the site based on the device dimensions so that the content is rearranged into fewer columns and/or hidden
  • Changing all fixed pixel width dimensions for the page contents into percent widths
  • Changing the font sizes, line heights, and margins based on the device dimensions
  • Vertically aligning the intro text responsively based on the page height instead of with a fixed margin-top
